Git Basics

Getting Started with Git

Git is a distributed version control system that allows developers to track changes in their codebase, collaborate with others, and manage different versions of their projects. Here are some fundamental concepts and commands to get you started:

Key Concepts

Understanding these key concepts will help you navigate Git more effectively:

  • Repository (repo)
    • The database of a projectâ€™s history. Create with git init or copy with git clone.

    This is where all your files are stored.

  • Working directory (working tree)
    • Your checked-out files. Edit here; inspect changes with git status and git diff. This is where you make changes to your files with your preferred editor.
  • Staging area (index)
    • Where changes are prepared before committing. Add files with git add.
  • HEAD
    • The current branch or commit reference; usually the latest commit on the checked-out branch.
  • Remote
    • A named connection to another repository (e.g., origin). Manage with git remote.
  • Origin
    • Default name of the remote created by git clone.
  • Upstream
    • The remote branch a local branch tracks. Set with git branch --set-upstream-to origin/main.
  • Merge conflict
    • Overlapping changes require manual resolution; mark resolved with git add and commit.
  • Detached HEAD
    • HEAD points directly to a commit, not a branch; commits can be lost if not referenced.
  • .gitignore
    • Patterns of files Git should not track.
  • Remote-tracking branch
    • Local read-only refs representing remote branches, e.g., origin/main.
  • Bare repository
    • A repo without a working tree, typically used on servers.
  • Blame
    • Show who last modified each line: git blame <file>.
  • Hooks
    • Scripts run on Git events, located in .git/hooks.


The Basic Commands

These are the most frequently used commands that you need to know to get started. More info below.

  • Clone
    • Create a local copy of a remote repository: git clone [repository URL].
  • Status
    • Check the status of your working directory and staging area: git status.
  • Add
    • Stage changes for the next commit: git add [file] or stage all changes with git add ..
  • Commit
    • Save staged changes with a message: git commit -m "Your commit message".
  • Push
    • Upload local commits to a remote repository: git push [remote] [branch].
  • Pull
    • Fetch and merge changes from a remote branch: git pull [remote] [branch].
  • Branch
    • Create a new branch for features or fixes: git branch [branch name].
  • Checkout
    • Switch branches or restore files: git checkout [branch name] or git checkout [file].
  • Merge
    • Combine changes from one branch into another: git merge [branch name].
  • Log
    • View commit history: git log.


init

Git init

Creates a new repository in the current or specified directory.
Used when you're starting a new project, or want to start version-controlling an existing project.

Usage: git init [options] [<directory>]

Options

  • -q, --quiet
    • Minimal output; suppresses init messages.
  • --bare
    • Creates a bare repo without a working directory. The root of the repo itself IS .git. Typical for servers/CI systems.
  • -b <name>, --initial-branch=<name>
    • Specifies the name of the first branch (overrides init.defaultBranch). Example: -b main.
  • --template=<directory>
    • Uses files from a template directory (hooks, attributes, exclude, etc.) during init.
  • --separate-git-dir=<directory>
    • Stores Git metadata in a separate location. The working directory then contains a .git pointer to that location.
  • --object-format=sha1|sha256
    • Chooses object format (hash algorithm) for the new repo. sha256 requires compatible tools and should only be used for entirely new repos.
  • --ref-format=files|reftable
    • Chooses storage format for references. files is default; reftable is a modern backend and may require a newer Git.
  • --shared[=<value>]
    • Sets sharing level/permissions (affects e.g. core.sharedRepository). Common values:
    â€¢ umask (default & respects system umask)
    â€¢ group (group-writable, good for teams)
    â€¢ all/world/everybody (read/write for everyone)
    â€¢ 0xxx (octal, e.g. 0770, 0660)
    Applies mainly to bare repos; in non-bare repos it mostly affects the .git contents.
  • <directory> (optional)
    • Location where the repo should be created. If omitted, the current directory is initialized.

Examples

  • Initialize in the current directory
    • git init
  • Initialize in a new directory with main branch "main"
    • git init -b main project
  • Bare repo for group sharing
    • git init --bare --shared=group /srv/git/my-projects.git
  • Separate Git metadata from working directory
    • git init --separate-git-dir=/var/lib/git/meta .
clone

Git clone

Creates a local copy of an existing Git repository. The clone contains all history, branches, and references from the source repository. Used to obtain a working copy from a remote repo (e.g. GitHub, GitLab, Bitbucket) or another local repo.

Usage: git clone [options] <repository> [<directory>]

Options

  • -q, --quiet
    • Suppresses progress reporting and diagnostic messages.
  • --bare
    • Creates a bare clone (no working directory). The root of the repo itself IS .git. Common for server-side repos.
  • --mirror
    • Creates a mirror clone. Implies --bare, and also clones all refs (including remote-tracking branches, notes, etc.). Useful for backups or server mirrors.
  • --branch <name>, -b <name>
    • Checks out the given branch instead of the default (usually HEAD of remoteâ€™s default branch).
  • --depth <n>
    • Creates a shallow clone with history truncated to the last n commits. Reduces download size.
  • --single-branch
    • Clones only the history leading to the tip of a single branch. Often combined with --branch or --depth.
  • --recurse-submodules[=yes|no|on-demand]
    • Initializes and clones submodules as well. Default is on-demand if configured, otherwise no.
  • --origin <name>, -o <name>
    • Uses the given name instead of origin for the remote.
  • --config key=value
    • Sets a configuration option in the new clone. Can be repeated multiple times.
  • <repository>
    • URL or path of the repo to clone. Can be HTTPS, SSH, Git protocol, or local path.
  • <directory> (optional)
    • Directory where the repo will be cloned. If omitted, Git uses the name of the repo.

Examples

  • Clone from HTTPS
    • git clone https://github.com/user/project.git
  • Clone with SSH
    • git clone git@github.com:user/project.git
  • Clone into a specific directory
    • git clone https://github.com/user/project.git mydir
  • Shallow clone with 1 commit
    • git clone --depth 1 https://github.com/user/project.git
  • Clone only a single branch
    • git clone --branch develop --single-branch https://github.com/user/project.git
  • Mirror an entire repo (for backup/server)
    • git clone --mirror https://github.com/user/project.git
status

Git status

Displays the state of the working directory and staging area.
Shows which changes are staged, unstaged, or untracked. Used frequently to check what will be included in the next commit.

Usage: git status [options]

Options

  • -s, --short
    • Outputs a short summary instead of the full descriptive format.
  • -b, --branch
    • Shows branch information along with status (helpful in short format).
  • --ignored
    • Also shows ignored files.
  • --untracked-files[=<mode>]
    • Controls how untracked files are shown. Modes: â€¢ no (do not show untracked files) â€¢ normal (default, shows untracked files and dirs) â€¢ all (shows individual untracked files inside dirs)

Examples

  • Check status in full format
    • git status
  • Compact summary with branch info
    • git status -sb
  • Hide untracked files
    • git status --untracked-files=no
  • Show ignored files as well
    • git status --ignored
add

Git add

Adds file contents to the staging area (index) in preparation for the next commit.
You use git add to tell Git which changes you want to include in the next commit. Does not affect the repository itself until git commit is run.

Usage: git add [options] [<pathspec>...]

Options

  • -A, --all
    • Stages all changes (tracked and untracked, including deletions).
  • -u, --update
    • Stages modifications and deletions of already tracked files, but not new files.
  • -p, --patch
    • Interactively select hunks of changes to stage.
  • -i, --interactive
    • Interactive mode for staging changes (menus and prompts).
  • -n, --dry-run
    • Shows what would be added without actually staging anything.
  • --ignore-removal
    • Ignores file deletions when staging changes.
  • --chmod=(+x|-x)
    • Changes the executable bit of staged files.
  • <pathspec>
    • Specifies the files or directories to add. Can include globs (e.g. *.js).

Examples

  • Stage a single file
    • git add file.txt
  • Stage all changes in the repo
    • git add -A
  • Stage only modified and deleted files (no new files)
    • git add -u
  • Stage all files in current directory
    • git add .
  • Interactively choose which hunks to add
    • git add -p
  • Mark a script as executable
    • git add --chmod=+x script.sh
commit

Git commit

Records the staged changes in the repository history as a new commit.
A commit creates a snapshot of the staging area, with a unique ID (hash), message, and metadata (author, timestamp).

Usage: git commit [options] [--] [<pathspec>...]

Options

  • -m <msg>
    • Specifies the commit message inline.
  • -a, --all
    • Stages all tracked changes (skips git add for modified/deleted files).
  • --amend
    • Replaces the last commit with a new one (includes staged changes, keeps history linear).
  • --no-edit
    • Used with --amend to keep the previous commit message without editing.
  • --dry-run
    • Shows what would be committed without actually creating a commit.
  • --allow-empty
    • Creates a commit even if no changes are staged.
  • --author="Name <email>"
    • Overrides the author for this commit.
  • <pathspec>
    • Optionally limit commit to specific files already staged.

Examples

  • Commit with a message
    • git commit -m "Add new feature"
  • Commit all tracked changes (skip git add)
    • git commit -a -m "Quick fix"
  • Amend the last commit (edit message or include new changes)
    • git commit --amend
  • Amend the last commit without changing the message
    • git commit --amend --no-edit
  • Commit with a custom author
    • git commit --author="Jane Doe <jane@example.com>" -m "Hotfix"
  • Create an empty commit (useful for CI triggers)
    • git commit --allow-empty -m "Trigger build"
