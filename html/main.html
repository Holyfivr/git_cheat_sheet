Git Basics

Getting Started with Git

Git is a distributed version control system that allows developers to track changes in their codebase, collaborate with others, and manage different versions of their projects. Here are some fundamental concepts and commands to get you started:

Key Concepts

Understanding these key concepts will help you navigate Git more effectively:

  • Repository (repo)
    • The database of a project’s history. Create with git init or copy with git clone.

    This is where all your files are stored.

  • Working directory (working tree)
    • Your checked-out files. Edit here; inspect changes with git status and git diff. This is where you make changes to your files with your preferred editor.
  • Staging area (index)
    • Where changes are prepared before committing. Add files with git add.
  • HEAD
    • The current branch or commit reference; usually the latest commit on the checked-out branch.
  • Remote
    • A named connection to another repository (e.g., origin). Manage with git remote.
  • Origin
    • Default name of the remote created by git clone.
  • Upstream
    • The remote branch a local branch tracks. Set with git branch --set-upstream-to origin/main.
  • Merge conflict
    • Overlapping changes require manual resolution; mark resolved with git add and commit.
  • Detached HEAD
    • HEAD points directly to a commit, not a branch; commits can be lost if not referenced.
  • .gitignore
    • Patterns of files Git should not track.
  • Remote-tracking branch
    • Local read-only refs representing remote branches, e.g., origin/main.
  • Bare repository
    • A repo without a working tree, typically used on servers.
  • Blame
    • Show who last modified each line: git blame <file>.
  • Hooks
    • Scripts run on Git events, located in .git/hooks.


The Basic Commands

These are the most frequently used commands that you need to know to get started. More info below.

  • Clone
    • Create a local copy of a remote repository: git clone [repository URL].
  • Status
    • Check the status of your working directory and staging area: git status.
  • Add
    • Stage changes for the next commit: git add [file] or stage all changes with git add ..
  • Commit
    • Save staged changes with a message: git commit -m "Your commit message".
  • Push
    • Upload local commits to a remote repository: git push [remote] [branch].
  • Pull
    • Fetch and merge changes from a remote branch: git pull [remote] [branch].
  • Branch
    • Create a new branch for features or fixes: git branch [branch name].
  • Checkout
    • Switch branches or restore files: git checkout [branch name] or git checkout [file].
  • Merge
    • Combine changes from one branch into another: git merge [branch name].
  • Log
    • View commit history: git log.


init

Git init

Creates a new repository in the current or specified directory.
Used when you're starting a new project, or want to start version-controlling an existing project.

Usage: git init [options] [<directory>]

Options

  • -q, --quiet
    • Minimal output; suppresses init messages.
  • --bare
    • Creates a bare repo without a working directory. The root of the repo itself IS .git. Typical for servers/CI systems.
  • -b <name>, --initial-branch=<name>
    • Specifies the name of the first branch (overrides init.defaultBranch). Example: -b main.
  • --template=<directory>
    • Uses files from a template directory (hooks, attributes, exclude, etc.) during init.
  • --separate-git-dir=<directory>
    • Stores Git metadata in a separate location. The working directory then contains a .git pointer to that location.
  • --object-format=sha1|sha256
    • Chooses object format (hash algorithm) for the new repo. sha256 requires compatible tools and should only be used for entirely new repos.
  • --ref-format=files|reftable
    • Chooses storage format for references. files is default; reftable is a modern backend and may require a newer Git.
  • --shared[=<value>]
    • Sets sharing level/permissions (affects e.g. core.sharedRepository). Common values:
    • umask (default & respects system umask)
    • group (group-writable, good for teams)
    • all/world/everybody (read/write for everyone)
    • 0xxx (octal, e.g. 0770, 0660)
    Applies mainly to bare repos; in non-bare repos it mostly affects the .git contents.
  • <directory> (optional)
    • Location where the repo should be created. If omitted, the current directory is initialized.

Examples

  • Initialize in the current directory
    • git init
  • Initialize in a new directory with main branch "main"
    • git init -b main project
  • Bare repo for group sharing
    • git init --bare --shared=group /srv/git/my-projects.git
  • Separate Git metadata from working directory
    • git init --separate-git-dir=/var/lib/git/meta .
clone

Git clone

Creates a local copy of an existing Git repository. The clone contains all history, branches, and references from the source repository. Used to obtain a working copy from a remote repo (e.g. GitHub, GitLab, Bitbucket) or another local repo.

Usage: git clone [options] <repository> [<directory>]

Options

  • -q, --quiet
    • Suppresses progress reporting and diagnostic messages.
  • --bare
    • Creates a bare clone (no working directory). The root of the repo itself IS .git. Common for server-side repos.
  • --mirror
    • Creates a mirror clone. Implies --bare, and also clones all refs (including remote-tracking branches, notes, etc.). Useful for backups or server mirrors.
  • --branch <name>, -b <name>
    • Checks out the given branch instead of the default (usually HEAD of remote’s default branch).
  • --depth <n>
    • Creates a shallow clone with history truncated to the last n commits. Reduces download size.
  • --single-branch
    • Clones only the history leading to the tip of a single branch. Often combined with --branch or --depth.
  • --recurse-submodules[=yes|no|on-demand]
    • Initializes and clones submodules as well. Default is on-demand if configured, otherwise no.
  • --origin <name>, -o <name>
    • Uses the given name instead of origin for the remote.
  • --config key=value
    • Sets a configuration option in the new clone. Can be repeated multiple times.
  • <repository>
    • URL or path of the repo to clone. Can be HTTPS, SSH, Git protocol, or local path.
  • <directory> (optional)
    • Directory where the repo will be cloned. If omitted, Git uses the name of the repo.

Examples

  • Clone from HTTPS
    • git clone https://github.com/user/project.git
  • Clone with SSH
    • git clone git@github.com:user/project.git
  • Clone into a specific directory
    • git clone https://github.com/user/project.git mydir
  • Shallow clone with 1 commit
    • git clone --depth 1 https://github.com/user/project.git
  • Clone only a single branch
    • git clone --branch develop --single-branch https://github.com/user/project.git
  • Mirror an entire repo (for backup/server)
    • git clone --mirror https://github.com/user/project.git
status

Git status

Displays the state of the working directory and staging area.
Shows which changes are staged, unstaged, or untracked. Used frequently to check what will be included in the next commit.

Usage: git status [options]

Options

  • -s, --short
    • Outputs a short summary instead of the full descriptive format.
  • -b, --branch
    • Shows branch information along with status (helpful in short format).
  • --ignored
    • Also shows ignored files.
  • --untracked-files[=<mode>]
    • Controls how untracked files are shown. Modes: • no (do not show untracked files) • normal (default, shows untracked files and dirs) • all (shows individual untracked files inside dirs)

Examples

  • Check status in full format
    • git status
  • Compact summary with branch info
    • git status -sb
  • Hide untracked files
    • git status --untracked-files=no
  • Show ignored files as well
    • git status --ignored
add

Git add

Adds file contents to the staging area (index) in preparation for the next commit.
You use git add to tell Git which changes you want to include in the next commit. Does not affect the repository itself until git commit is run.

Usage: git add [options] [<pathspec>...]

Options

  • -A, --all
    • Stages all changes (tracked and untracked, including deletions).
  • -u, --update
    • Stages modifications and deletions of already tracked files, but not new files.
  • -p, --patch
    • Interactively select hunks of changes to stage.
  • -i, --interactive
    • Interactive mode for staging changes (menus and prompts).
  • -n, --dry-run
    • Shows what would be added without actually staging anything.
  • --ignore-removal
    • Ignores file deletions when staging changes.
  • --chmod=(+x|-x)
    • Changes the executable bit of staged files.
  • <pathspec>
    • Specifies the files or directories to add. Can include globs (e.g. *.js).

Examples

  • Stage a single file
    • git add file.txt
  • Stage all changes in the repo
    • git add -A
  • Stage only modified and deleted files (no new files)
    • git add -u
  • Stage all files in current directory
    • git add .
  • Interactively choose which hunks to add
    • git add -p
  • Mark a script as executable
    • git add --chmod=+x script.sh
commit

Git commit

Records the staged changes in the repository history as a new commit.
A commit creates a snapshot of the staging area, with a unique ID (hash), message, and metadata (author, timestamp).

Usage: git commit [options] [--] [<pathspec>...]

Options

  • -m <msg>
    • Specifies the commit message inline.
  • -a, --all
    • Stages all tracked changes (skips git add for modified/deleted files).
  • --amend
    • Replaces the last commit with a new one (includes staged changes, keeps history linear).
  • --no-edit
    • Used with --amend to keep the previous commit message without editing.
  • --dry-run
    • Shows what would be committed without actually creating a commit.
  • --allow-empty
    • Creates a commit even if no changes are staged.
  • --author="Name <email>"
    • Overrides the author for this commit.
  • <pathspec>
    • Optionally limit commit to specific files already staged.

Examples

  • Commit with a message
    • git commit -m "Add new feature"
  • Commit all tracked changes (skip git add)
    • git commit -a -m "Quick fix"
  • Amend the last commit (edit message or include new changes)
    • git commit --amend
  • Amend the last commit without changing the message
    • git commit --amend --no-edit
  • Commit with a custom author
    • git commit --author="Jane Doe <jane@example.com>" -m "Hotfix"
  • Create an empty commit (useful for CI triggers)
    • git commit --allow-empty -m "Trigger build"
log

Git log

Displays the commit history of the repository.
By default, it shows commits in reverse chronological order (most recent first), including commit hash, author, date, and message.

Usage: git log [options] [<revision range>] [<path>...]

Options

  • --oneline
    • Shows each commit on a single line (abbreviated hash + message).
  • --graph
    • Draws an ASCII graph of branch/merge history alongside commits.
  • --decorate
    • Adds branch and tag names to commit listings.
  • --stat
    • Shows changed files and line counts for each commit.
  • -p, --patch
    • Shows the actual changes (diffs) introduced by each commit.
  • --since=<date>, --after=<date>
    • Limits commits to those more recent than the given date.
  • --until=<date>, --before=<date>
    • Limits commits to those older than the given date.
  • --author=<pattern>
    • Only shows commits by a given author.
  • --grep=<pattern>
    • Searches commit messages for the given pattern.
  • -n <number>
    • Limits the number of commits shown (e.g. -n 5).

Examples

  • Show commit history
    • git log
  • Show 5 most recent commits
    • git log -n 5
  • Compact one-line history
    • git log --oneline
  • Visualize branch history with graph
    • git log --oneline --graph --decorate
  • Show commits by a specific author
    • git log --author="Alice"
  • Search commit messages for "fix"
    • git log --grep="fix"
  • Show commits made in the last 2 weeks
    • git log --since="2 weeks ago"
  • Show commits that modified a specific file
    • git log -- path/to/file.txt
diff

Git diff

Shows the differences between commits, branches, the staging area, and the working directory.
Commonly used to review changes before staging, before committing, or between branches/commits.

Usage: git diff [options] [<commit>] [<commit>] [--] [<path>...]

Options

  • --staged / --cached
    • Shows differences between the staged changes (index) and the last commit.
  • --name-only
    • Shows only the names of changed files.
  • --name-status
    • Shows file names along with their status (Added, Modified, Deleted).
  • --color
    • Forces colored output (enabled by default in most terminals).
  • -U<n>
    • Controls the number of context lines shown around differences (default is 3).
  • commit1 commit2
    • Shows changes between two commits, branches, or tags.
  • <path>
    • Limits the diff to specific files or directories.

Examples

  • Show unstaged changes in working directory
    • git diff
  • Show staged changes (ready to commit)
    • git diff --staged
  • Show changes between two commits
    • git diff commit1 commit2
  • Show changes in a specific file
    • git diff HEAD~1 HEAD -- file.txt
  • Show only file names that changed
    • git diff --name-only
  • Show file names and status (A/M/D)
    • git diff --name-status
branch

Git branch

Lists, creates, or deletes branches in a repository.
Branches are pointers to commits, used to isolate work on features or fixes.

Usage: git branch [options] [<branch-name>] [<start-point>]

Options

  • -a, --all
    • Lists both local and remote branches.
  • -r, --remotes
    • Lists only remote-tracking branches.
  • -d <branch>
    • Deletes the specified branch (safe delete, refuses if not merged).
  • -D <branch>
    • Force deletes the branch, even if unmerged.
  • -m <new-name>
    • Renames the current branch to the given name.
  • -M <new-name>
    • Forces renaming of the current branch (overwrites if exists).
  • --show-current
    • Shows the name of the currently checked-out branch.
  • <branch-name>
    • Creates a new branch pointing at the current commit or a given start point.

Examples

  • List all local branches
    • git branch
  • List all branches, local and remote
    • git branch -a
  • Create a new branch called "feature"
    • git branch feature
  • Delete a branch safely
    • git branch -d feature
  • Force delete a branch
    • git branch -D old-branch
  • Rename the current branch
    • git branch -m new-name
  • Show current branch
    • git branch --show-current
checkout

Git checkout

Switches branches or restores working tree files.
Commonly used to move between branches, check out commits, or restore file contents.

Usage: git checkout [options] [<branch>|<commit>] [--] [<pathspec>...]

Options

  • -b <branch>
    • Creates and checks out a new branch.
  • -B <branch>
    • Creates or resets a branch to a specific commit and checks it out.
  • --detach
    • Checks out a commit directly, putting HEAD into "detached" state.
  • -- <pathspec>
    • Checks out specific files or directories from a branch or commit into the working directory.

Examples

  • Switch to an existing branch
    • git checkout develop
  • Create and switch to a new branch
    • git checkout -b feature
  • Switch to a specific commit (detached HEAD)
    • git checkout abc1234
  • Restore a file from a branch
    • git checkout main -- file.txt
  • Reset a branch to a specific commit
    • git checkout -B feature abc1234
merge

Git merge

Combines changes from one branch into another.
Most commonly used to merge a feature branch into the current branch. If changes conflict, Git will pause and require manual resolution before completing the merge.

Usage: git merge [options] <branch>...

Options

  • --no-ff
    • Always creates a merge commit, even if the merge could be resolved with a fast-forward.
  • --ff-only
    • Refuses to merge unless it can be resolved as a fast-forward (no merge commit).
  • --squash
    • Combines all changes from the merged branch into a single commit, without recording merge history.
  • --abort
    • Aborts the merge process and resets the working tree to the state before the merge started.
  • --continue
    • Continues a merge after conflicts have been resolved.
  • --no-edit
    • Uses the default merge commit message without opening an editor.
  • --commit / --no-commit
    • Controls whether Git automatically creates a merge commit. With --no-commit, changes are staged but not committed.

Examples

  • Merge a feature branch into the current branch
    • git merge feature-branch
  • Force a merge commit (disable fast-forward)
    • git merge --no-ff feature-branch
  • Only merge if fast-forward is possible
    • git merge --ff-only feature-branch
  • Squash commits into one (without merge history)
    • git merge --squash feature-branch
  • Abort a merge after conflicts
    • git merge --abort
  • Continue merge after resolving conflicts
    • git merge --continue
rebase

Git rebase

Reapplies commits from one branch on top of another base commit.
Used to maintain a linear project history by moving or combining commits.

Usage: git rebase [options] [<upstream>] [<branch>]

Options

  • -i, --interactive
    • Interactive mode to reorder, squash, or edit commits.
  • --continue
    • Continues the rebase after resolving conflicts.
  • --abort
    • Aborts the rebase and restores the branch to its original state.
  • --skip
    • Skips the current commit during a conflict resolution.
  • --onto <newbase>
    • Rebases commits onto a different base commit.

Examples

  • Rebase current branch onto main
    • git rebase main
  • Interactive rebase last 3 commits
    • git rebase -i HEAD~3
  • Abort an ongoing rebase
    • git rebase --abort
  • Continue after fixing conflicts
    • git rebase --continue
  • Rebase onto another branch
    • git rebase --onto develop feature
stash

Git stash

Temporarily saves uncommitted changes (both staged and unstaged) so you can work on something else and reapply them later.

Usage: git stash [options] [push]

Options

  • save <message>
    • (Deprecated) Save changes with an optional description.
  • push -m <message>
    • Saves changes with a custom message.
  • list
    • Shows all stashed states.
  • apply [<stash>]
    • Applies the stash but keeps it in the stash list.
  • pop [<stash>]
    • Applies the stash and removes it from the list.
  • drop [<stash>]
    • Removes a stash without applying it.
  • clear
    • Removes all stashed entries.

Examples

  • Stash current changes
    • git stash
  • Stash with a message
    • git stash push -m "WIP: login form"
  • List all stashes
    • git stash list
  • Apply most recent stash
    • git stash apply
  • Pop (apply and remove) the most recent stash
    • git stash pop
  • Drop a specific stash
    • git stash drop stash@{1}
reset

Git reset

Resets the current HEAD to a specified state.
Can modify the staging area and/or working directory depending on the mode used.

Usage: git reset [options] [<commit>] [--] [<paths>...]

Options

  • --soft
    • Moves HEAD to the commit, keeps staging area and working directory unchanged.
  • --mixed (default)
    • Moves HEAD and resets the staging area, but keeps working directory changes.
  • --hard
    • Resets HEAD, staging area, and working directory. All changes are lost.
  • <commit>
    • Specifies the commit to reset to. Can be SHA, branch name, HEAD~N, etc.
  • <paths>
    • If paths are specified, only those files are reset from the index.

Examples

  • Unstage a file (keep working changes)
    • git reset file.txt
  • Reset to last commit, keep changes staged
    • git reset --soft HEAD~1
  • Reset to last commit, keep changes in working directory
    • git reset --mixed HEAD~1
  • Reset to last commit, discard all changes
    • git reset --hard HEAD~1
  • Reset index for a specific file
    • git reset HEAD file.txt
fetch

Git fetch

Downloads objects and refs from another repository.
Unlike git pull, it does not modify your working directory or merge changes. It only updates remote-tracking branches.

Usage: git fetch [options] [<remote>] [<refspec>...]

Options

  • -a, --all
    • Fetches all remotes.
  • -p, --prune
    • Removes remote-tracking branches that no longer exist on the remote.
  • --tags
    • Fetches all tags in addition to branches.
  • --depth <n>
    • Fetches only the last n commits from the history (shallow fetch).
  • --force
    • Forces updating local refs even if it would overwrite changes.

Examples

  • Fetch from default remote (origin)
    • git fetch
  • Fetch from a specific remote
    • git fetch upstream
  • Fetch all remotes
    • git fetch --all
  • Fetch and prune deleted remote branches
    • git fetch --prune
  • Fetch only last 5 commits
    • git fetch --depth 5
pull

Git pull

Fetches changes from a remote repository and immediately integrates them into the current branch.
By default, it combines git fetch + git merge. With certain options, it can use rebase instead of merge.

Usage: git pull [options] [<remote>] [<branch>]

Options

  • --rebase
    • Rebases current branch on top of the fetched branch instead of merging.
  • --no-rebase
    • Explicitly disables rebasing (default is merge).
  • --ff-only
    • Only updates if it can be done as a fast-forward. Aborts otherwise.
  • --all
    • Fetches all remotes and integrates them.
  • <remote>
    • Name of the remote to pull from (default is origin).
  • <branch>
    • Specific branch to pull. Defaults to the remote tracking branch for the current branch.

Examples

  • Pull changes from origin’s default branch
    • git pull
  • Pull a specific branch
    • git pull origin develop
  • Pull with rebase instead of merge
    • git pull --rebase
  • Pull only if fast-forward possible
    • git pull --ff-only
push

Git push

Uploads local commits, branches, and tags to a remote repository.
Used to share changes with collaborators by updating the remote with your local work.

Usage: git push [options] [<remote>] [<branch>]

Options

  • -u, --set-upstream
    • Sets the upstream (tracking) reference for the branch you’re pushing.
  • -f, --force
    • Forces the push even if it will overwrite remote history (dangerous).
  • --force-with-lease
    • Safer alternative to force; refuses to overwrite if the remote has new commits you don’t have locally.
  • --tags
    • Pushes all local tags to the remote.
  • --delete
    • Deletes a remote branch or tag.
  • <remote>
    • Name of the remote (default is origin).
  • <branch>
    • Branch to push. If omitted, pushes the current branch.

Examples

  • Push current branch to origin
    • git push origin main
  • Set upstream and push
    • git push -u origin develop
  • Force push branch
    • git push --force origin feature
  • Push all tags
    • git push --tags
  • Delete a remote branch
    • git push origin --delete feature
tag

Git tag

Creates, lists, deletes, or verifies tags.
Tags are references to specific commits, often used for marking release points (e.g., v1.0).

Usage: git tag [options] [<tagname>] [<commit>]

Options

  • -a <tagname>
    • Create an annotated tag with a message, author, and date.
  • -m <message>
    • Message to use for annotated tag.
  • -d <tagname>
    • Delete a tag.
  • -l, --list
    • Lists tags, optionally with a pattern.
  • -s <tagname>
    • Create a signed tag (requires GPG setup).
  • <commit>
    • The commit the tag should reference. Defaults to HEAD.

Examples

  • List all tags
    • git tag
  • Create a lightweight tag
    • git tag v1.0
  • Create an annotated tag
    • git tag -a v1.0 -m "Release version 1.0"
  • Tag a specific commit
    • git tag v1.1 abc1234
  • Delete a tag
    • git tag -d v1.0
remote

Git remote

Manages tracked repositories (remotes).
Lets you add, remove, rename, and inspect connections to other repositories.

Usage: git remote [options] [command] [<name>] [<url>]

Options

  • -v
    • Shows URLs of remotes when listing.
  • add <name> <url>
    • Adds a new remote.
  • remove <name> / rm <name>
    • Removes a remote.
  • rename <old> <new>
    • Renames an existing remote.
  • show <name>
    • Displays information about a specific remote.
  • set-url <name> <newurl>
    • Changes the URL for an existing remote.

Examples

  • List remotes
    • git remote -v
  • Add a new remote
    • git remote add origin https://github.com/user/project.git
  • Rename a remote
    • git remote rename origin upstream
  • Remove a remote
    • git remote remove upstream
  • Change remote URL
    • git remote set-url origin git@github.com:user/project.git
